## SECTION 1 ÔÇö PROJECT IDENTITY

This system is a high-availability, enterprise-grade data ingestion and normalization backend designed to synchronize and process messaging data from the Turn.io (WhatsApp Business Solution Provider) Data Export API. It solves the problem of reliable data capture from highly asynchronous and sometimes unstable third-party APIs while maintaining a historical archive and real-time state for a multi-tenant messaging platform. The system is used by business operations and automation services to provide response-time analytics, journey tracking, and audit-ready logs for WhatsApp-based customer interactions. The end goal is to maintain a 100% consistent, queryable, and normalized PostgreSQL database reflecting every message, status update, and contact interaction occurring across all registered WhatsApp business numbers.

## SECTION 4 ÔÇö SYSTEM ARCHITECTURE

**Containers and processes:**
The system runs in two primary containerized processes:
1. **Ingestion Service**: Hosts the FastAPI app (`app/main.py`). It handles inbound webhooks and runs background ingestion tasks (`MessageWorker`, `ContactWorker`). It exposes port 8000 (API) and 9102 (Prometheus Metrics).
2. **Normalization Service**: A worker process (`normalization/main.py`) that polls the `webhook_events` table for raw data and transforms it into structured rows. It has no public ports.

**Data flow ÔÇö end to end:**
```
Turn.io (Webhook) Ôćĺ Ingestion (/webhooks/turn) Ôćĺ webhook_events (processed=FALSE) Ôćĺ Normalization Service Ôćĺ [messages, statuses, contacts, chats, chat_state] (processed=TRUE)
Turn.io (API Pull) Ôćĺ Ingestion (Worker) Ôćĺ webhook_events (processed=FALSE) Ôćĺ Normalization Service Ôćĺ [messages, statuses, contacts, chats, chat_state] (processed=TRUE)
```
- **Ingestion**: Real-time (webhook) or Scheduled (API Pull) capture of raw JSON into `webhook_events`.
- **Durable Queue**: `webhook_events` acts as a write-ahead log/queue.
- **Normalization**: Transformation of JSON into relational rows. Every event is processed as a separate transaction.
- **Storage**: Canonical tables are updated via Upserts (`ON CONFLICT`).

**Two separate data entry points:**
1. **Webhook handler**: Real-time. Idempotent based on `(provider, external_event_id)`. Ensures immediate capture.
2. **Ingestion workers**: Scheduled periodic pulls. Ensure data completeness for periods where webhooks may have failed or been dropped by network issues.

## SECTION 5 ÔÇö DATABASE SCHEMA SUMMARY

- `webhook_events`: Primary landing zone for raw JSON. PK: `id`. writes: Ingestion. reads: Normalization.
- `contacts`: Central WhatsApp identity store. PK: `id`, `uuid`. writes: Normalization.

- `statuses`: Delivery state trace. PK: `(message_id, status)`. writes: Normalization.

- `ingestion_state`: Worker checkpoints (id=1: Msg, id=2: Contact). PK: `id`. writes: Ingestion.
- `normalization_state`: Worker checkpoint for normalizer. PK: `id`. writes: Normalization.
- `numbers`: Registered Business Numbers. PK: `id`.
- `normalization_failures`: Error repository for malformed events. PK: `id`. writes: Normalization.

## SECTION 6 ÔÇö INGESTION SERVICE (Digital_Ocean_backend/app)

**Startup sequence**: FastAPI lifespan starts Ôćĺ Init DB pool via `create_shared_pool()` Ôćĺ Init `IngestionManager` with shared pool Ôćĺ Call `manager.start()` Ôćĺ `ContactWorker` task launched immediately Ôćĺ `MessageWorker` task launched after 10s stagger.

**IngestionManager**: Orchestrates workers. Owns `AsyncRateLimiter` (token bucket: rate=20, burst=20). Owns `TurnClient` (messages) and `TurnContactClient` (contacts). monitors tasks via `_on_worker_done` callback.

**MessageWorker**:
- **Incremental Loop**: Every 30s. Fetches from Turn Export API. Checkpoint in `ingestion_state` id=1.
- **Recovery Scheduler**: Once every 24h. Scans last 3 days for gaps.
- **Audit Scheduler**: Once every 7 days. Scans last 30 days for integrity.
- **Metrics Server**: Port 9102. Tracks lag (timestamp difference) and throughput.

**ContactWorker**: Every 60 minutes. Performs full Startup Backfill on restart. Uses parallel chunked fetching via `TurnContactClient`. Writes to `webhook_events` as `event_type='contact'`. Checkpoint in `ingestion_state` id=2.

**Webhook handler (`/webhooks/turn`)**: Extracts ID using `_extract_event_id` (Priority: message.id > status.id > Hash). Writes raw JSON to `webhook_events` with `processed=FALSE` and `ON CONFLICT DO NOTHING`.

**Rate limiter**: Shared token bucket across all clients. Configured for 20 req/s to stay safely within Turn's 600/30s limit.

**Known issues (Forensic Audit Unresolved)**:
- **C1 (Duplicate Pools)**: Stale `db_pool` import in `main.py` causes manager to create a redundant pool and readiness path to use `None`. (Needs Audit 1.6/Section 11 fix).
- **C2 (Checkpoint Durability)**: MessageWorker advances checkpoint *before* `executemany` flush is durable. Crash here = data skip. (`messages_worker.py:433-434`).
- **C3 (ID Collision)**: Message and Status events for same message share `external_event_id`. `webhook_events` unique index drops one. (`main.py:121-126`).
- **C4 (Auth)**: No webhook signature verification. Publicly accessible. (`main.py:133-257`).
- **C5 (Deadlock)**: Contact worker consumer failures swallow exceptions and can hang producer on full queue. (`contacts_worker.py:202-205`).
- **H5 (Readiness False Positive)**: Readiness returns `ready: true` even if workers are stopped. (`main.py:88-95`).

## SECTION 7 ÔÇö NORMALIZATION SERVICE (Digital_Ocean_backend/normalization)

**Purpose**: Offline transformation of raw Turn.io JSON into standard SQL schema. Provides reliability and re-processability.

**Startup sequence**: Load config Ôćĺ Init `db_writer.init_pool` Ôćĺ Idempotent schema check via `state_manager.setup_schema` Ôćĺ Load `normalization_state` id=1 Ôćĺ Apply `START_FROM_ID` if virgin state Ôćĺ Start `worker.heartbeat_loop` Ôćĺ Start `worker.run_worker`.

**Event routing**:
- `message`: Triggers 8-step message pipeline.
- `status`: Triggers delivery/read status updates.
- `contact`: Contextually processed during message flow.

**Processing pipeline ÔÇö message events (8 steps):**
1. **Guard**: Validate `_vnd.v1` block and `payload.id` (`worker.py:105-108`).
2. **Contact resolution**: `db_writer.upsert_contact` (Preserves name, conflict on `uuid`).
3. **Chat resolution**: `db_writer.upsert_chat` (Conflict on `uuid`). Retries with `assigned_to_uuid=NULL` on FK violation.
4. **Campaign resolution**: `_resolve_campaign` lookup via `journey_uuid`.
5. **Message insert**: `db_writer.insert_message` (Conflict on `external_id` DO NOTHING).
6. **Status upsert**: `db_writer.upsert_status` (Uses `parsers.validate_status` enum guard).
7. **Chat state update**: `db_writer.upsert_chat_state` (Updates unread, `last_inbound_at`, `last_outbound_at`).
8. **Attachment insert**: `db_writer.insert_attachment` (Media metadata).
9. **Chat metadata**: `db_writer.update_last_message` (Conditional update based on timestamp).

**Processing pipeline ÔÇö status events**:
Look up original message ID via `messages.external_id` Ôćĺ `db_writer.process_status_event_db` Ôćĺ Upsert to `statuses` table.

**Checkpoint model**: Tracks maximum processed `webhook_events.id`. Checkpoint moved ONLY after a full batch of transactions successfully completes.

**Transaction model**: One database transaction per event ID.

**Failure handling**: Catches `Exception`, writes record to `normalization_failures` (traceback + payload), and proceeds to next event.

**Heartbeat and shutdown**: Heartbeat emits throughput metrics every 60s. Shutdown uses `asyncio.Event` to stop loop cleanly on `SIGTERM`.

**Files and their responsibilities:**
- `main.py`: Entry point, signals, and process lifecycle.
- `worker.py`: Batch loop manager, transactional event orchestration.
- `db_writer.py`: All SQL queries. `init_pool`, `upsert_contact`, `upsert_chat`, `insert_message`, `upsert_status`, `upsert_chat_state`, `insert_attachment`, `update_last_message`, `insert_failure`.
- `parsers.py`: Pure extraction logic. `normalize_phone`, `parse_timestamp`, `validate_direction`, `validate_status`, `parse_contact`, `parse_chat`, `parse_message`, `parse_chat_state`.
- `models.py`: Dataclasses: `ContactRow`, `ChatRow`, `MessageRow`, `StatusRow`, `ChatStateRow`.

## SECTION 12 ÔÇö QUICK REFERENCE

**Key Identifiers:**
- **External Event ID**: `payload.id`. Maps to `webhook_events.external_event_id` and `messages.external_id`.
- **Contact Identity**: `payload._vnd.v1.chat.contact_uuid`. Used to link messages to contacts.
- **Conversation Identity**: `payload._vnd.v1.chat.uuid`. Crucial for chat grouping.

**Checkpoint Tables:**
- `ingestion_state`: Tracks progress of API pulls. `id=1` is messages, `id=2` is contacts.
- `normalization_state`: Tracks processed `webhook_events.id`. Advances only after successful batch commit.

**Processor Flag:**
- `webhook_events.processed`: Set to `FALSE` on insert. System currently uses `normalization_state` for progress, but this flag is intended for future queryability.

**Service Owners:**
- **Ingestion**: Owns `webhook_events`, `ingestion_state`.
- **Normalization**: Owns `messages`, `contacts`, `chats`, `statuses`, `chat_state`, `normalization_failures`, `normalization_state`.
