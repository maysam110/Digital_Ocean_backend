SECTION 3 — END-TO-END DATA FLOW
================================================================================

A) Webhook Processing Flow (Turn.io -> API -> DB)

1. Request arrival
- HTTP POST `/webhooks/turn` (`main.py:133-134`)
- JSON body parsed via `await request.json()` (`main.py:135`)

2. Payload parsing and ID extraction
- Event type read from `payload["type"]` (`main.py:136`).
- Stable external ID derived via `_extract_event_id` (`main.py:105-130`):
  a) `message.id`
  b) `status.id`
  c) fallback deterministic hash

3. Idempotent raw webhook write
- Inserts into `webhook_events` with `ON CONFLICT (provider, external_event_id) DO NOTHING` (`main.py:141-146`).

4. Message event branch (`event_type == "message"`)
- Checks existing `messages.external_id` (`main.py:158-161`).
- Inserts message row if absent (`main.py:168-184`).

5. Status event branch (`event_type == "status"`)
- Resolves message row by `messages.external_id=status.id` (`main.py:195-202`).
- Inserts status row with payload timestamp parse fallback (`main.py:206-255`).

6. Error handling behavior
- Message insert exceptions are caught/logged (`main.py:185-190`) and request still returns success.
- Status insert exceptions are not locally caught; DB errors bubble and produce request failure.

7. Critical idempotency conflict
- Status and message events may share same external ID (`_extract_event_id` status path `main.py:121-126` vs message path `115-119`).
- Since unique key excludes `event_type`, one of the two event types can be dropped in `webhook_events` (`main.py:145`, schema `Digital_Ocean_Schema.sql:1716`).


B) Messages Ingestion Flow

Startup sequence
1. FastAPI lifespan starts (`main.py:37-60`).
2. DB pool created via `connect_db` (`main.py:53`, `db.py:12-24`).
3. Manager created and `start` task launched (`main.py:56-58`).
4. Manager creates shared limiter + Turn clients + workers (`manager.py:55-86`).
5. Contact worker starts immediately (`manager.py:88-94`).
6. Message worker starts after stagger delay (`manager.py:101-105`, `153-159`).

Checkpoint retrieval
- Message checkpoint read from `ingestion_state.id=1` (`messages_worker.py:174-183`).
- Bootstrap row created with `STARTUP_FROM_DATE` if missing (`messages_worker.py:112-114`).

Overlap logic
- Each incremental cycle sets `from_ts = checkpoint - 2s` (`messages_worker.py:563-565`).
- This intentionally re-fetches overlap window.

Cursor creation
- `TurnClient._create_data_export_cursor()` called with date window (`base.py:191-232`).

Page iteration
- `_fetch_messages_single_cursor` loops pages using paging next cursor (`base.py:419-499`).

Retry behavior
- Cursor creation retries: max 10 for 429/network (`base.py:218-255`).
- Page fetch retries: max 5 for network, 429, and 5xx (`base.py:272-310`).
- Cursor expiry recovery: re-create from last offset, max 10 (`base.py:425-440`).

Consumer batching
- Producer pushes tuples into bounded queue (`messages_worker.py:361-427`).
- `NUM_CONSUMERS` batch consumers flush DB (`messages_worker.py:442-445`).

DB writes
- `INSERT INTO webhook_events ... ON CONFLICT DO NOTHING` (`messages_worker.py:519-523`).

ON CONFLICT logic
- Message worker dedupes by unique `(provider, external_event_id)`.

Checkpoint update
- Progress updates during fetch every 1000 scanned (`messages_worker.py:433-434`).
- Final update after cycle if `max_ts` found (`messages_worker.py:577-580`).
- Update guarded to move forward only (`messages_worker.py:223-231`).

Crash behavior
- If crash after progress checkpoint update but before batch flushes complete, checkpoint may advance past undurable records (data skip risk).

Restart behavior
- On restart, ingestion resumes from persisted checkpoint minus overlap; overlap is only 2s, so large undurable window can be skipped.


C) Contacts Ingestion Flow (differences vs messages)

Core differences
- Contacts worker performs full startup backfill each process start (`contacts_worker.py:371-399`).
- Data is written as event type `contact` into `webhook_events` via UPSERT (`contacts_worker.py:317-322`).

Mutable record logic
- `ON CONFLICT DO UPDATE` overwrites payload and resets `processed=FALSE` (`contacts_worker.py:319-321`).
- This is explicitly designed to refresh mutable contact profiles.

Payload refresh rules
- Existing contact event payload for same `(provider, external_event_id)` is replaced.

`updated_at` vs `inserted_at`
- Checkpoint candidate timestamp priority: `updated_at` then `inserted_at` (`contacts_worker.py:224-225`).
- Same priority used in contact API page recovery state (`base.py:823-825`, `942-944`).

Checkpoint semantics
- Uses `ingestion_state.id=2` (`contacts_worker.py:107`).
- Advances only when higher timestamp exists (`contacts_worker.py:134-140`).

SECTION 4 — CHECKPOINT & STATE MODEL
================================================================================

4.1 ingestion_state usage
- `id=1`: messages checkpoint (`messages_worker.py:178`, `225-231`)
- `id=2`: contacts checkpoint (`contacts_worker.py:107`, `134-140`)
- Table schema: `Digital_Ocean_Schema.sql:480-484`

4.2 Forward-only guarantee
- Enforced by SQL predicate `last_external_timestamp < $1` in both workers (`messages_worker.py:229-231`, `contacts_worker.py:138-140`).

4.3 Regression prevention
- Message worker rejects naive/future/outlier timestamps (`messages_worker.py:197-213`).
- Future clamping for extracted message timestamps (`base.py:1130-1133`).

4.4 Overlap buffer mechanics
- Messages overlap: 2 seconds (`messages_worker.py:57`, `563-565`).
- Contacts overlap: 2 seconds (`contacts_worker.py:52`, `419-421`).
- Purpose: reduce boundary misses at exact timestamp edges.

4.5 Mid-cycle process death behavior
- Messages: checkpoint can move during producer progress (`messages_worker.py:433-434`) before all queued writes are committed; crash here can skip data.
- Contacts: checkpoint updates only after full cycle return (`contacts_worker.py:428-433`), safer against mid-cycle skip.

4.6 DB write failure behavior
- Messages: batch failure raises and fails cycle (`messages_worker.py:535-537`, `452-461`), but prior progress checkpoint updates may already be persisted.
- Contacts: consumer failures logged/swallowed (`contacts_worker.py:202-205`), potentially causing hang without explicit cycle failure.

4.7 API partial failure behavior
- Messages API has bounded retries and cursor regeneration (`base.py:424-460`).
- Contacts API chunk failures move to sequential fallback (`base.py:841-845`, `890-962`).

4.8 Bootstrap behavior
- Initial checkpoint bootstrap from `STARTUP_FROM_DATE` via insert-if-missing (`messages_worker.py:112-114`, `contacts_worker.py:97-99`).

4.9 Validation against requested properties

- Infinite re-fetch loop:
  - Cursor-expiry loops are bounded (`base.py:416-417`, `427-431`, `800-801`, `927-930`).
  - Service loops are intentionally perpetual schedulers.
  - Verdict: No unbounded cursor-expiry loop; expected perpetual polling loops exist by design.

- Timestamp drift:
  - Future timestamps are rejected/clamped (`messages_worker.py:202-204`, `base.py:1130-1133`, `contacts_worker.py:128-130`).
  - Verdict: Mostly controlled.

- Skipped window risk:
  - Present in messages due progress checkpoint before durability (`messages_worker.py:433-434`).
  - Verdict: Not guaranteed safe.

- Checkpoint race condition:
  - Single-instance forward-only SQL guard is robust.
  - Multi-instance deployments still contend with concurrent writers and duplicated ingestion load.

SECTION 5 — RATE LIMITER & CONCURRENCY
================================================================================

5.1 AsyncRateLimiter math correctness

Implementation (`base.py:107-149`): token bucket.
- Refill: `tokens = min(burst, tokens + elapsed * rate/period)` (`base.py:133-136`)
- Acquire condition: `tokens >= requested` (`base.py:140-142`)
- Wait time when empty: `wait = (needed * period)/rate` (`base.py:144-145`)

Configured by manager:
- `rate=20`, `period=1.0`, `burst=50` (`manager.py:57`)

Derived limits:
- Sustained throughput = 20 requests/second.
- Short burst = 50 immediate requests.
- Envelope over T seconds: `max_requests(T) = burst + rate*T` (initial window).
- For 30s: `50 + 20*30 = 650` requests theoretical, which is higher than documented Turn limit 600/30s.

5.2 Burst behavior and sliding-window risk
- There is a potential 30-second overshoot by up to 50 requests at cold start/high contention due burst configuration.
- This likely manifests as additional 429 responses, not hard deadlock.

5.3 Lock correctness
- Active implementation in `base.py` sleeps outside lock (`base.py:127-148`) and avoids lock hold during await.
- Shared limiter is single process-local object (`manager.py:57`, passed to both clients `59-63`).

5.4 Shared limiter across workers
- Message and contact clients share one limiter in one process (`manager.py:57-63`).
- No distributed limiter across processes/replicas.

5.5 Potential starvation
- No strict fairness queue in limiter; starvation is possible under heavy contention but bounded by cooperative scheduling.

5.6 Event-loop blocking risks
- No blocking `time.sleep` found in active async paths.
- CPU-heavy `json.dumps` in hot loops can still add event-loop pressure (`messages_worker.py:513-514`, `contacts_worker.py:310-311`).

5.7 Task cancellation handling
- Message pipeline handles cancellation and gathers canceled tasks (`messages_worker.py:452-461`).
- Contact pipeline does similar at orchestration level (`contacts_worker.py:276-285`).
- Manager stop does not await worker task termination (`manager.py:164-197`).

5.8 Validation against requested checks
- No unbounded concurrency: Mostly true (bounded semaphores/queues), but chunk/task counts are large and configurable.
- No blocking calls inside async: true for major I/O paths.
- No deadlock risk: false (contact consumer-failure queue deadlock path exists).
- No memory explosion: bounded but high; large queue settings can still exceed practical memory limits.

SECTION 9 — PERFORMANCE ANALYSIS
================================================================================

9.1 Max queue memory usage (estimated)

Configured capacities:
- Messages ingestion queue: 10,000 items.
- Contacts fetch queue: 50,000 items.
- Contacts ingest queue: 30,000 items.

If average serialized payload size is:
- 1 KB/event: raw payload bytes ~90 MB total across queues.
- 2 KB/event: raw payload bytes ~180 MB.

Python object overhead (dict/tuple/string wrappers) typically multiplies raw payload memory by ~2x to ~3x.
Expected peak process memory contribution from queues can therefore exceed ~200 MB and plausibly reach ~500+ MB in heavy backlog situations.

9.2 API throughput ceiling

Limiter parameters (`rate=20/s`, `burst=50`):
- Sustained request ceiling: 20 req/s combined across message + contact workers.
- Startup burst: up to 50 immediate requests.

Messages API request payload includes `limit=100` (`base.py:214`), so upper-bound message fetch throughput (API-side) is approximately:
- `20 req/s * 100 msg/req = 2,000 msg/s` before DB/storage bottlenecks.

Contacts page size is API-defined (not explicitly set), so throughput is variable.

9.3 DB write throughput

- Message worker uses `executemany` with batch size 1000 (`messages_worker.py:55`, `517-525`).
- Contact worker also uses batch size 1000 (`base.py:72`, `185`, `315-324`).

Effective throughput depends on DB latency, conflict rate, and payload size. In conflict-heavy windows, logical insert gain can be low while write load remains high.

9.4 Worst-case ingestion lag

- Normal target loop period: 30s messages (`messages_worker.py:58`, `606`).
- Under repeated failures, lag can grow unbounded over time.
- Watchdog recovery triggers when lag > 900s (`messages_worker.py:65`, `591-597`).

9.5 Recovery time from large backlog

Let:
- `B = backlog messages`
- `R = effective durable writes/sec`

Recovery time ≈ `B / R` seconds.
Examples:
- B=10,000,000 and R=1,000/s -> ~2.78 hours
- B=10,000,000 and R=300/s -> ~9.26 hours

9.6 Backfill scaling behavior

Messages
- Large ranges split into 7-day sequential chunks (`base.py:343-367`).

Contacts
- Range split into N chunks (default 24) with parallel fetch tasks (`base.py:741`, `765-773`, `861-865`).
- Failed chunks retried sequentially with global timeout (`base.py:890-905`).

SECTION 13 — FINAL VERDICT
================================================================================

Production Ready?
- No (current state is not production-safe for high-integrity ingestion).

Must Fix Before Production
1. Fix `db_pool` stale import/rebinding and enforce single shared pool (`main.py:8`, `57`; `db.py:13-14`).
2. Remove/redo progress checkpoint advancement so checkpoint only reflects durable DB writes (`messages_worker.py:433-434`).
3. Resolve `webhook_events` uniqueness collision between message and status event IDs (`main.py:121-126`, `143-146`; schema `1716`).
4. Add webhook authentication/signature validation (`main.py:133-257`).
5. Fix contact pipeline failure propagation/deadlock path (`contacts_worker.py:202-205`, `238-241`, `259-260`).
6. Make status insert idempotent under duplicate callbacks (`main.py:224-255`, schema `1786`).

Can Wait (but should be planned)
- Duplicate schema indexes/FKs cleanup.
- Runtime DDL migration normalization.
- Dependency pinning and deployment health gating.
- Repository artifact cleanup.

Architecturally Strong
- Clear separation between API and ingestion modules.
- Strong retry scaffolding and cursor-expiry handling.
- Bounded queue designs and explicit lag monitoring.

Architecturally Fragile
- Checkpoint durability semantics.
- Shared-table key design across heterogeneous event types.
- Process-local assumptions (limiter/state) under scale-out.
- Pool lifecycle coupling and readiness signaling.

Is ingestion core enterprise-grade?
- Partially. Scheduling, retries, and monitoring are advanced, but correctness gaps (checkpoint durability and deadlock paths) prevent enterprise-grade reliability.

Is webhook layer robust?
- No. It lacks authentication and has conflict/duplicate handling weaknesses for status events.

Is scaling safe?
- Single instance only, with caveats.
- Horizontal scaling is unsafe without distributed rate limiting, singleton coordination, and checkpoint leadership controls.

Is data loss possible?
- Yes.
- Primary mechanism: checkpoint advanced before durable writes in message ingestion path.
